# Streamlit 회원가입 · 로그인 · 권한 관리 시스템

Streamlit 단일 애플리케이션으로 구현한 인증/인가 시스템입니다. 별도의 백엔드 서버 없이, SQLite와 Streamlit의 세션 상태만으로 로그인, 회원가입 승인, 권한(role) 관리, 보안 방어 로직, 감사 로그, 계정 간 메시지 기능까지 구현했습니다.

## 핵심 특징

- **가입 승인제**: 회원가입 즉시 로그인되지 않고, 관리자가 승인해야 로그인 가능
- **권한(role) 요청/승인**: 일반 유저가 관리자 권한을 요청하면, 관리자가 검토 후 승인/거절
- **다층 방어**: 이메일 기준 로그인 잠금, IP 기준 로그인 잠금, 회원가입 스팸 방지, 이미지 캡차, 캡차 반복 시도 방지
- **중복(동시) 로그인 차단**: 이미 로그인된 계정으로 추가 로그인 시도 시 차단하고 관리자에게 알림
- **감사 로그**: 로그인 성공/실패, 잠금, 가입 승인/거절, 권한 변경, 로그아웃 등 주요 이벤트 전부 기록
- **새로고침 유지**: 세션 토큰을 URL 쿼리 파라미터에 저장해 페이지를 새로고침해도 로그인이 풀리지 않음
- **계정 간 메시지**: 폴링(자동 새로고침) 기반의 간단한 인앱 채팅

## 아키텍처

```
브라우저
   │
   ▼
index.py  ── 진입점. 로그인 게이트(require_login)를 호출하고,
   │         로그인 성공 시 st.navigation으로 페이지 라우팅
   │
   ├──▶ auth.py         (로그인 안 됨) 로그인/회원가입 폼, 캡차, 세션 발급
   │
   └──▶ webpages/pages/*.py  (로그인 됨) 기능 페이지들
              │
              ▼
        accountdb.py  ── 모든 DB 접근을 전담하는 단일 계층
              │            (세션, 잠금 카운터, 알림, 감사 로그, 권한 요청)
              ▼
        account.db (SQLite)
```

**계층을 나눈 이유**: `index.py`는 "로그인 여부를 확인하고 어떤 화면을 보여줄지 결정"하는 역할만, `auth.py`는 "인증 관련 UI와 로직", `accountdb.py`는 "DB 읽기/쓰기"만 담당하도록 분리했습니다. 이렇게 하면 나중에 로그인 화면 디자인만 바꾸고 싶을 때 `auth.py`만, DB 스키마를 바꾸고 싶을 때 `accountdb.py`만 건드리면 되고, 각 파일이 무슨 역할을 하는지 파일명만 보고 파악할 수 있습니다.

## 파일 구조

| 파일 | 역할 |
|---|---|
| `index.py` | 진입점, 로그인 게이트 호출, 페이지 라우팅, 사이드바 |
| `auth.py` | 로그인/회원가입 폼, 캡차, 세션 발급/복구, 로그아웃 |
| `accountdb.py` | DB 스키마, 세션·잠금·알림·감사 로그·권한 요청 관련 모든 함수 |
| `create_admin.py` | 관리자 계정을 안전하게(해시 처리해서) 만드는 CLI 스크립트 |
| `webpages/pages/approvals.py` | (관리자) 회원가입 승인/거절 |
| `webpages/pages/role_request.py` | (일반 유저) 관리자 권한 요청 |
| `webpages/pages/role_requests_admin.py` | (관리자) 권한 요청 승인/거절 |
| `webpages/pages/security_alerts.py` | (관리자) 계정/IP 잠금, 동시 로그인 시도 등 보안 알림 확인 |
| `webpages/pages/audit_log.py` | (관리자) 전체 감사 로그 조회 |
| `webpages/pages/messages.py` | 계정 간 인앱 메시지 |

## 데이터베이스 테이블

| 테이블 | 용도 |
|---|---|
| `users` | 계정, 해시된 비밀번호, role, 가입 상태(pending/approved/rejected), 마지막 로그인 |
| `sessions` | 새로고침 유지 및 중복 로그인 판별용 세션 토큰 (해시로만 저장, 만료시각 포함) |
| `login_attempts` / `ip_login_attempts` | 이메일/IP 기준 로그인 실패 카운트 및 잠금 |
| `signup_attempts` | IP 기준 회원가입 시도 카운트 및 잠금 |
| `captcha_attempts` | IP 기준 캡차 실패 카운트 및 잠금 |
| `notifications` | 관리자에게 보내는 알림 (가입 대기, 권한 요청, 계정/IP 잠금, 동시 로그인 시도) |
| `role_requests` | 권한 변경 요청 이력 |
| `audit_log` | 모든 주요 이벤트의 감사 로그 |
| `messages` | 계정 간 메시지 |

## 보안 설계 요약

| 위협 | 대응 |
|---|---|
| 평문 비밀번호 저장 | `werkzeug`의 scrypt 해시로 저장 |
| 세션 토큰 탈취 | DB에는 토큰 원본이 아닌 SHA-256 해시만 저장, 7일 후 자동 만료 |
| 무차별 대입(같은 계정) | 이메일 기준 5회 실패 시 5분 잠금 |
| 무차별 대입(여러 계정 순회) | IP 기준 20회 실패 시 15분 잠금 |
| 자동화 봇의 반복 요청 | 로컬 생성 이미지 캡차 (로그인 3회 실패 후, 회원가입은 항상) |
| 캡차 자체를 스크립트로 반복 시도 | IP 기준 캡차 5회 실패 시 10분 차단 |
| 계정 대량 생성(스팸 가입) | IP 기준 회원가입 5회/30분 제한 |
| 승인 안 된 계정의 무단 사용 | 가입 즉시 로그인 불가, 관리자 승인 후에만 로그인 가능 |
| 탈취된 계정을 동시에 사용 | 이미 활성 세션이 있으면 추가 로그인 차단 + 관리자 알림 + 감사 로그 |
| 사후 추적 불가 | 모든 주요 이벤트를 `audit_log`에 기록 |

## 알려진 한계 (Streamlit 아키텍처 특성)

- **진짜 실시간 푸시 불가**: Streamlit은 요청-응답 기반이라 서버가 클라이언트에 즉시 알림을 보낼 수 없습니다. 메시지 기능은 `streamlit-autorefresh`로 3초 폴링을 흉내낸 것입니다.
- **httpOnly 쿠키 미지원**: Streamlit 코어에는 서버가 `Set-Cookie` 헤더를 직접 제어하는 공식 API가 없어, 세션 유지에 URL 쿼리 파라미터 방식을 사용했습니다 (토큰은 해시로만 저장해 위험을 완화).
- **IP 확인이 환경에 따라 다름**: `st.context.ip`는 로컬 환경이나 일부 배포 환경에서 `None`을 반환할 수 있습니다. `auth.py`에 데모/로컬용 `"unknown"` 대체값이 들어있으니, 실서비스 배포 시 해당 줄을 검토하세요.

## 알려진 한계 (SQLite 아키텍처 특성)

SQLite는 서버가 아니라 파일 하나이기 때문에, MySQL/PostgreSQL 같은 서버형 DB와는 다른 종류의 구조적 한계가 있습니다. `GRANT` 기반 권한 부여가 안 된다는 점 외에도 아래 사항들이 있습니다.

- **DB 자체에 접속 인증이 없음**: `app.db` 파일에 파일시스템 접근 권한만 있으면, 애플리케이션 코드를 거치지 않고 `sqlite3 app.db` 명령으로 누구나 전체 내용을 열람할 수 있습니다. 아이디/비밀번호로 DB 접속을 막는 개념 자체가 없습니다. → 파일 권한(`chmod 600`)과 실행 계정 분리로만 방어 가능합니다.
- **저장 데이터가 기본적으로 암호화되지 않음**: 비밀번호는 해시 처리했지만, 이메일 주소나 채팅 메시지 내용 같은 다른 컬럼들은 파일을 열면 평문 그대로 보입니다. 컬럼 단위/파일 단위 암호화가 필요하면 SQLCipher 같은 별도 확장이 있어야 합니다 (기본 SQLite엔 없음).
- **DB 차원의 감사 로그가 없음**: 서버형 DB는 "누가 언제 어떤 쿼리를 날렸는지" DB 자체에서 기록해주는 기능이 있는 경우가 많은데, SQLite는 그런 기능이 없습니다. 지금 프로젝트의 `audit_log` 테이블은 DB가 아니라 **애플리케이션 코드가 스스로 기록**한 것이라, 코드에 로그 남기는 걸 빠뜨린 경로가 있으면 그 행동은 추적이 안 됩니다.
- **동시 쓰기 시 파일 전체 잠금**: SQLite는 행(row) 단위가 아니라 파일 단위로 쓰기 잠금을 겁니다. 사용자가 많아지면 여러 요청이 동시에 쓰려고 할 때 `database is locked` 오류가 날 수 있습니다. 지금 규모(내부 도구)에는 문제 없지만, 사용자가 많아지면 서버형 DB로 옮기는 걸 고려해야 하는 이유이기도 합니다.

**정리**: 지난번 논의한 "역할 기반 세밀한 권한 부여가 안 된다"는 문제와, 위 항목들("접속 자체에 대한 인증/암호화/감사가 없다")은 서로 다른 문제입니다. 둘 다 근본적으로는 "SQLite는 서버가 아니라 파일"이라는 같은 원인에서 나오지만, 실무에서는 각각 다른 대응이 필요합니다 (전자는 서버형 DB 전환, 후자는 파일 권한/암호화/백업 정책).

## 동작상 참고사항 — 중복 로그인 차단

- 같은 계정으로 이미 로그인된 상태에서 또 로그인을 시도하면 차단되고, 관리자에게 계정/시도 IP와 함께 알림이 갑니다.
- 이 판단 기준은 **DB의 `sessions` 테이블에 그 계정의 유효한(만료 안 된) 세션이 남아있는지**입니다. 그래서 **로그아웃은 반드시 "그 계정의 모든 세션"을 지워야** 합니다 — 지금 켜져 있는 브라우저의 세션 하나만 지우면, 예전에 만들어진 다른 세션이 남아서 재로그인 자체가 막혀버립니다. `logout()`이 `delete_all_sessions_for_user()`를 호출하는 이유입니다.
- 로그아웃 버튼을 안 누르고 그냥 브라우저 창을 닫은 경우엔 세션이 DB에 그대로 살아있어서(최대 7일), 같은 기기로 다시 로그인해도 "이미 로그인 되어있습니다"로 막힐 수 있습니다 — 의도된 동작입니다.
